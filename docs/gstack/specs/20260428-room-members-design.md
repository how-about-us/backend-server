# Design: 방 멤버 목록 조회 API

Status: APPROVED

## Problem Statement

협업 앱에서 "누가 이 방에 참여하고 있고, 누가 지금 접속 중인지"를 보여주는 것은 기본 인프라 기능. 이 기능 자체가 사용자를 끌어오지는 않지만, 없으면 협업 경험이 성립하지 않음.

접속 상태를 가시화하면 social accountability의 토대를 마련할 수 있음 — "누가 참여 중인지 보이면 기여하고 싶어진다"는 가설.

## Cross-Model Perspective

Claude subagent 독립 분석 결과:
- 핵심 문제는 도구 분산이 아니라 **노동 비대칭(labor asymmetry)**. 기여를 가시적으로 만들지 않으면 올인원으로 합쳐도 짐이 옮겨갈 뿐.
- 멤버 목록 + 접속 상태는 social accountability의 첫 레이어. 장기적으로는 activity attribution(누가 뭘 추가했는지)이 더 중요.
- REST+STOMP 이중 채널의 동기화 갭 지적이 있었으나, 3-8명 규모에서 실익이 낮고 기존 아키텍처 일관성을 위해 현재 패턴 유지로 결정.

## Approaches Considered

### Approach A: Minimal Viable — DB 조회 + Redis 접속 상태 조합

기존 `findByRoom_IdAndRole()` + EntityGraph로 멤버 조회, `getOnlineUserIds()`로 접속 상태 조합. 기존 코드 최대 활용. Effort: S.

### Approach B: Ideal Architecture — 역할별 정렬 + 확장 구조

Approach A에 HOST 우선, 접속 중 MEMBER, 오프라인 MEMBER 순 정렬 추가. 향후 activity attribution 확장 가능 구조. Effort: M.

### Approach C: 접속 유저 조회 API 통합

features.md의 "방 멤버 목록 조회"와 "현재 접속 중인 유저 조회" 두 미구현 항목을 하나의 API로 통합. 3-8명 규모에서 분리할 실익 없음.

## Recommended Approach

**A+C 결합**: 멤버 목록 API 하나에 접속 상태를 포함. 별도 접속 유저 API는 만들지 않음.

이유:
- 소규모 그룹에서 멤버 목록 + 접속 상태를 분리할 실익이 거의 없음
- features.md의 두 미구현 항목을 한 번에 해결
- 기존 코드(repository, presence service, authorization)를 그대로 재활용
- 정렬은 프론트엔드에서 처리하는 게 더 유연

## Open Questions

- 향후 activity attribution(누가 뭘 추가했는지) 기능 추가 시 이 API에 포함할지 별도 API로 할지
- 멤버 수가 많아졌을 때 페이지네이션 필요 여부 (현재 3-8명 규모에서는 불필요)
