# Design: 멤버 추방 + 방 나가기 API

Generated by /office-hours on 2026-04-29
Branch: feature/roomMembers
Status: APPROVED

## Problem Statement

여행 계획 협업 서비스에서 승인된 멤버를 내보내야 하는 상황이 발생할 수 있다. 현재 추방 기능이 없으면 방 전체를 삭제하고 새로 만들어야 하는데, 이 경우 보관함·일정·채팅 히스토리가 전부 소실된다. 한 명의 문제 멤버 때문에 모든 멤버의 협업 데이터가 인질이 되는 구조.

근거는 팀 내부 예상 시나리오 기반. 승인 후 사회적 상황 변화(여행에서 빠지는 사람, 갈등 등)로 멤버를 내보내야 하는 케이스를 대비한다.

## Demand Evidence

- 직접적인 사용자 요청은 아직 없음 (pre-product 단계)
- 입장 승인/거절 시스템과 초대 코드 재발급이 이미 있어 사전 방어는 충분
- 추방이 없을 때의 유일한 대안은 "방 삭제 후 재생성" → 데이터 전량 소실이라는 치명적 비용
- 그룹 관리의 기본 기능으로서 사용자 기대에 부합

## Status Quo

추방 기능 없이 문제 멤버 발생 시:
1. 방 삭제 후 새로 생성 → 보관함, 일정, 채팅 히스토리 전량 소실
2. 그냥 참기 → 협업 경험 저하
3. 초대 코드 재발급으로 추가 진입 방지 가능하나, 이미 들어온 멤버는 처리 불가

## Constraints

- 3-8명 규모의 친구/지인 그룹 서비스 → 과한 보안/모더레이션 불필요
- 기존 아키텍처: Spring Boot 4 + Java 21, PostgreSQL, MongoDB (채팅), Redis (접속 상태), WebSocket + STOMP
- `room_members` 테이블의 role: HOST / MEMBER / PENDING
- 기존 패턴: reject (PENDING 멤버 삭제), approve (PENDING → MEMBER), 시스템 메시지 (AFTER_COMMIT 이벤트)

## Premises

1. **HOST만 추방 가능. HOST 자신은 추방 불가.**
2. **추방 시 room_members 레코드만 삭제.** 보관함·일정·채팅 데이터는 유지. `bookmarks.added_by`는 단순 Long 값으로 users 테이블에 대한 FK 제약이 없으므로 room_members 삭제 시 무결성 문제 없음.
3. **추방된 멤버가 초대 코드로 재입장 시 PENDING 상태.** 기존 입장 승인 플로우 재활용. HOST 재승인 필요.
4. **추방 시 시스템 메시지 브로드캐스트.** 기존 입장 승인 시스템 메시지 패턴 재활용.
5. **WebSocket 이벤트 전달, 세션 끊기는 클라이언트 책임.** 서버 세션 강제 종료는 3-8명 규모에서 과잉이고 레이스 컨디션 위험.
6. **"방 나가기" 기능과 함께 설계.** 둘 다 room_members 삭제. 차이점은 authorization (kick=HOST, leave=본인)과 비즈니스 룰 (HOST는 leave 불가).

## Cross-Model Perspective

Claude subagent 독립 분석 결과:
- kick을 "기존 방 삭제라는 핵무기 대신 외과적 수술을 가능하게 하는 최소한의 모더레이션 기능"으로 평가
- 전제 2 (데이터 유지)에 도전: 추방된 멤버의 보관함 항목이 "유령 존재"를 만든다는 지적. 검증 방법으로 "출시 후 추방 직후 HOST가 보관함 삭제하는 비율 30% 이상이면 전제 재검토" 제안
- 3-8명 친구 그룹 맥락에서 과한 우려로 판단, 전제 2 유지 결정

## Approaches Considered

### Approach A: Kick Only (Minimal)
- 추방만 구현, 방 나가기와 WebSocket 처리는 나중에
- Effort: S / Risk: Low

### Approach B: Kick + Leave + 이벤트 전달
- kick과 leave를 같은 패턴으로 함께 구현
- WebSocket으로 추방/탈퇴 이벤트를 전달하되 세션 끊기는 클라이언트가 처리
- Effort: M / Risk: Low

### Approach C: Full Governance (서버 세션 강제 끊기 포함)
- 모든 것 + 서버에서 STOMP 세션 직접 종료
- Effort: L / Risk: Medium

## Recommended Approach

**Approach B: Kick + Leave + 이벤트 전달**

이유:
- kick과 leave는 동일한 패턴(room_members 삭제)이라 함께 만드는 게 효율적
- WebSocket은 이벤트 전달만, 세션 종료는 클라이언트 → 레이스 컨디션 회피
- 기존 reject/approve 패턴, 시스템 메시지 패턴을 최대한 재활용
- 3-8명 친구 그룹에 적합한 복잡도

## Open Questions

- 추방 사유(reason) 필드 필요 여부 → 3-8명 규모에서는 과잉으로 판단, 향후 필요 시 추가
- 출시 후 추방 직후 HOST가 추방된 멤버의 보관함 항목을 삭제하는 비율 추적 → 30% 이상이면 "추방 시 콘텐츠 삭제 옵션" 검토
- HOST가 방을 나가고 싶을 때의 처리: 현재는 방장 위임이 보류 상태이므로 HOST는 방 삭제만 가능
- 추방된 멤버의 기존 STOMP 구독은 클라이언트가 끊을 때까지 유지됨 (known limitation)
- 추방/탈퇴 후 채팅 히스토리 접근 불가 (requireActiveMember 체크). 향후 읽기 전용 접근이 필요하면 별도 검토

## Success Criteria

- HOST가 특정 MEMBER를 추방할 수 있다
- HOST 자신은 추방할 수 없다
- HOST가 아닌 사용자는 추방할 수 없다
- MEMBER가 자발적으로 방을 나갈 수 있다
- HOST는 방을 나갈 수 없다 (방장 위임 미구현)
- 추방/탈퇴 시 시스템 메시지가 채팅에 표시된다
- 추방/탈퇴 시 WebSocket 이벤트가 전달된다
- 추방된 멤버가 초대 코드로 재입장 시 PENDING 상태가 된다
- 보관함·일정·채팅 데이터는 추방/탈퇴와 무관하게 유지된다

## Dependencies

- 기존 `RoomAuthorizationService` — HOST 권한 확인
- 기존 `RoomMemberRepository` — 멤버 조회/삭제
- 기존 시스템 메시지 패턴 (`MessageService` + `MemberApprovedMessageListener` 패턴)
- 기존 `RoomPresenceService` — Redis 접속 상태 관리
- 기존 STOMP 메시지 브로드캐스트 인프라
