# review-code-against-docs 스킬 설계

**목표:** PR 올리기 전, `git diff main` 기준으로 변경된 코드가 프로젝트 문서(컨벤션·기능 스펙·ERD)와 일치하는지 순차적으로 검사한다.

## 배경

- `checking-md-conflicts`는 MD 파일 간 충돌만 검사하며 코드↔문서 연결은 하지 않는다.
- `code-review` 스킬은 PR 단위 전체 리뷰로 무겁다.
- 컨벤션 위반과 스펙 불일치는 성격이 달라 리뷰 때 함께 놓치기 쉽다.

이 스킬은 위 두 스킬의 틈새를 채우는 **PR 전 경량 셀프 체크** 도구다.

## 핵심 설계 원칙

**스킬에 규칙을 하드코딩하지 않는다.**

컨벤션·스펙·ERD 규칙은 각각의 문서(`CONTRIBUTING.md`, `docs/ai/features.md`, `docs/ai/erd.md`)가 단일 진실 공급원(Source of Truth)이다. 스킬은 런타임에 해당 문서를 직접 읽고 규칙을 추출하여 검사한다. 문서가 바뀌면 스킬을 수정하지 않아도 자동으로 반영된다.

## 범위

**포함**

- `CONTRIBUTING.md` Code Convention 섹션 기반 컨벤션 위반 탐지
- `docs/ai/features.md` 기능 스펙 대조
- `docs/ai/erd.md` 테이블/컬럼 ↔ 엔티티 불일치 탐지

**제외**

- `docs/ai/decisions/` — 인프라 수준 결정 기록이 많아 코드 레벨 검사와 연결이 약함
- 변경되지 않은 파일 전체 스캔
- 자동 수정 — 보고만 하고 수정은 사람이 판단

## 트리거 조건

- PR 올리기 전 수동 실행
- `finishing-a-development-branch` 스킬 흐름에서 자연스럽게 연결 가능

## 설계: 순차 체크리스트

### Step 0. diff 수집

```bash
git diff main --name-only
```

변경된 `.java` 파일 목록을 추출한다. 변경 파일이 없으면 "변경 없음"으로 종료한다.

### Step 1. 컨벤션 검사

1. `CONTRIBUTING.md`를 읽는다.
2. `Code Convention` 섹션 전체(하위 섹션인 `테스트` 포함)에서 규칙을 추출한다.
3. Step 0에서 얻은 변경 파일들을 읽고, 추출한 규칙 각각에 대해 위반 여부를 판단한다.
   - `src/main/` 파일은 일반 코드 규칙 기준으로 검사한다.
   - `src/test/` 파일은 테스트 규칙 기준으로 검사한다.

스킬은 "어떤 규칙이 있는지"를 알지 못한다. 문서에서 읽은 내용이 검사 기준의 전부다.

### Step 2. 스펙 정합성 검사

1. `docs/ai/features.md`를 읽는다.
2. 변경된 파일의 도메인(패키지명 기준)과 관련된 기능 섹션을 찾는다.
3. 해당 섹션의 기능 설명과 실제 구현 내용을 대조한다.
4. 구현이 완료됐는데 `[ ]`로 남아 있는 항목이 있으면 `[주의]`로 표시한다.

### Step 3. ERD 정합성 검사

`entity/` 경로 하위 파일이 변경된 경우에만 실행한다.

1. `docs/ai/erd.md`를 읽는다.
2. 변경된 엔티티와 대응하는 테이블 정의를 찾는다.
3. 컬럼명, 타입, 제약조건을 `@Column`, 필드명, JPA 어노테이션과 대조한다.

엔티티 변경이 없으면 `[건너뜀]`으로 표시한다.

## 보고서 형식

태그는 네 가지로 통일한다.

| 태그 | 의미 |
|------|------|
| `[통과]` | 문제 없음 |
| `[위반]` | 컨벤션 또는 스펙 불일치 발견 |
| `[주의]` | 명확하지 않아 사람이 직접 확인 필요 |
| `[건너뜀]` | 해당 파일 없어 검사 생략 |

```
## Code-Docs Review

### 1. 컨벤션
- [위반] PlaceController: @RequiredArgsConstructor 미사용 (CONTRIBUTING.md: "일반적인 생성자 주입은 @RequiredArgsConstructor를 우선 사용한다")
- [위반] GlobalExceptionHandler: 삼항 연산자 (line 19, 28) (CONTRIBUTING.md: "삼항 연산자는 사용하지 않는다")
- [통과] 코드 depth

### 2. 스펙 정합성
- [통과] 장소 검색 API — features.md 스펙과 일치
- [주의] features.md의 장소 검색 항목이 [ ] 상태 — 구현 완료 여부 확인 필요

### 3. ERD 정합성
- [건너뜀] 엔티티 변경 없음

2개 이슈 발견
```

위반 보고 시 해당 규칙이 어느 문서의 어느 문장에서 왔는지 함께 인용한다.

## 기존 스킬과의 관계

| 스킬 | 역할 | 이 스킬과의 관계 |
|------|------|-----------------|
| `checking-md-conflicts` | MD ↔ MD 충돌 검사 | 역할 겹치지 않음 |
| `code-review` | PR 단위 전체 리뷰 | 이 스킬은 그것의 경량 전처리 |
| `finishing-a-development-branch` | 브랜치 완료 정리 | 이 스킬을 내부 흐름에 연결 가능 |
